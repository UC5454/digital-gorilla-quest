# AI社員設計ガイド

## AI社員とは？

AI社員とは、Claude CodeのCLAUDE.mdに詳細な設定を書き込むことで、特定の役割・性格・スキルを持った「AI版の従業員」を作ること。
単なるチャットボットとの違いは、**記憶を持ち、役割を理解し、自律的に判断できる**点にあります。

## CLAUDE.mdの設計構造

AI社員のCLAUDE.mdは以下のセクションで構成します：

### 1. identity（魂）

AI社員の最も基本的な情報。名前・役割・一人称を定義。

```markdown
<identity>
[名前]は[組織名]の[役職]。一人称は「[私/僕/etc]」。
[1-2行のアイデンティティ説明]
</identity>
```

**ポイント：**
- 名前は覚えやすく、役割が想像できるものが理想
- 一人称は性格を反映する重要な要素
- 短く、でも「この人はどんな存在か」が伝わるように

### 2. role（役割）

責任範囲と判断基準を明確に定義。

```markdown
<role>
<responsibility>
## 責任範囲
| 領域 | 内容 | 境界 |
|------|------|------|
| [領域名] | [やること] | [やらないこと] |
</responsibility>

<decision_criteria>
## 判断の優先順位
1. [最優先の判断基準]
2. [次の基準]
</decision_criteria>

<when_uncertain>
## 迷った時の指針
- [どうすればいいか]
</when_uncertain>
</role>
```

**ポイント：**
- 「境界」を明確にすることで、AI社員が迷わない
- 判断基準に優先順位をつけることで、矛盾する指示にも対処できる

### 3. persona（人格）

性格・コミュニケーションスタイル・口調を定義。

```markdown
<persona>
<personality>
## パーソナリティ
- [性格特性1]
- [性格特性2]
</personality>

<communication_style>
## コミュニケーションスタイル
- 一人称: [私/僕/etc]
- 基本トーン: [敬語/カジュアル/etc]
- 口調の例:
  - ○「[良い例]」
  - ×「[悪い例]」
</communication_style>
</persona>
```

**ポイント：**
- 具体例（○と×）を入れると、AIが口調をブレなく再現できる
- NGワードを定義すると、不自然な表現を防げる

### 4. skills（スキル）

得意なこと・不得意なこと・ワークフローを定義。

```markdown
<skills>
## 得意なこと
- [スキル1]
- [スキル2]

## 不得意なこと
- [スキル1]

## ワークフロー
1. [手順1]
2. [手順2]
</skills>
```

### 5. startup（起動時の動作）

AI社員が起動した時に最初にやることを定義。

```markdown
<startup>
起動時に以下のファイルを読み込むこと:
- @CurrentTask.md（自分の進行中タスク）
- @INBOX.md（受信メッセージの確認）
- @daily-logs/（当日分のログ）
</startup>
```

### 6. memory_actions（記憶の書き込み）

対話の最後にやる記録作業を定義。

```markdown
<memory_actions>
毎回の対話終了時に、以下の手順でdaily-logsに記録すること:
1. daily-logs/YYYY-MM-DD.md に記録
2. 今日の作業内容・学んだこと・次のアクションを記載
</memory_actions>
```

### 7. confirmation_rules（確認ルール）

どこまで自律的に動くか、どこで確認を取るかのルール。

```markdown
<confirmation_rules>
- ファイル操作: 確認なしで即実行
- 外部送信: 必ず確認を取る
- 重大判断: [特定の人]に相談
</confirmation_rules>
```

### 8. inbox_rules（連携ルール）

他のAI社員との連携方法を定義。

```markdown
<inbox_rules>
## メッセージの送り方
相手のINBOX.mdに追記する。
書式: 【[自分の名前]より】メッセージ内容
</inbox_rules>
```

## 設計のベストプラクティス

1. **具体的に書く**: 曖昧な指示は曖昧な行動を生む
2. **例を入れる**: ○×の具体例があるとAIの再現精度が上がる
3. **境界を明確に**: 「やること」だけでなく「やらないこと」も書く
4. **段階的に育てる**: 最初から完璧を目指さず、使いながら調整する
5. **フィードバックループ**: daily-logsで振り返り、CLAUDE.mdを改善する

## よくある失敗パターン

| 失敗 | 原因 | 対策 |
|------|------|------|
| 口調がブレる | 具体例が少ない | ○×例を5つ以上入れる |
| 判断を間違える | 判断基準がない | decision_criteriaを詳しく書く |
| 勝手に動く | 境界が曖昧 | confirmation_rulesを明確に |
| 記憶が残らない | memory_actionsがない | 記録ルールを定義する |
